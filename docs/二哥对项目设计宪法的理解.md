我觉得你现在其实已经把**真正的目标**说出来了，这也是我觉得 V5 最大的问题。

> **Project-Lens 不是一个 Career Intelligence Layer。**
>
> **它首先应该是一个 Project Understanding Engine。**
>
> 求职只是它的第一个应用（Skill）。

这两个定位会导致整个架构完全不同。

---

# 我建议先写一个「Project-Lens 宪法（Constitution）」

以后所有设计都必须满足它，否则一律删掉。

例如：

---

# Project-Lens Constitution v1

## 第一条：唯一目标

> **降低 Agent 理解大型项目的成本。**

不是生成简历。

不是分析 JD。

不是模拟面试。

这些全部属于上层 Skill。

Lens 永远只负责：

> **把代码转换成 Agent 更容易理解的知识。**

---

## 第二条：所有输出必须来源于项目

Lens 的任何输出都必须能够追溯到：

* Code
* Git
* Test
* Benchmark
* Documentation

不能凭空推断。

例如：

✅

```
Redis 被引入于：

commit xxx

原因：

fix session bottleneck
```

而不是

```
这是缓存优化能力
```

因为这是解释。

---

## 第三条：Lens 不理解职业

职业属于 Agent。

例如：

```
JD：

需要 Redis
```

Lens 不应该说：

```
你符合80%
```

Lens 应该回答：

```
项目中找到：

Redis

ioredis

pub/sub

session

对应 Evidence：

...

```

Agent 自己判断是不是满足 JD。

---

## 第四条：Lens 提供的是 Exploration，不是 Answer

这是我觉得 V5 最缺的一条。

Lens 不应该只会 Search。

应该帮助 Agent：

探索项目。

例如：

Agent：

> 我想理解 Authentication。

Lens：

```
Authentication

↓

JWT

↓

Middleware

↓

Permission

↓

User Service

↓

Database

```

这不是回答。

这是 Exploration。

这一点我觉得比 Search 更重要。

---

## 第五条：任何工具都必须回答一个问题

这是判断工具是否应该存在的方法。

例如：

Tool：

```
search_evidence()
```

回答：

> 哪些事实相关？

成立。

---

Tool：

```
verify_statement()
```

回答：

> 这句话是真的吗？

成立。

---

Tool：

```
extract_capabilities()
```

回答：

> Redis 是不是缓存能力？

这个已经开始解释。

违反 Constitution。

---

## 第六条：Lens 不产生职业 Artifact

例如：

下面这些：

Resume

Interview

Career Plan

全部不是 Lens。

Lens 只产生：

Project Snapshot

Knowledge Graph

Evidence

Fact

Decision Trace

---

## 第七条：Skill 建立在 Lens 上

例如：

```
Interview Skill

↓

调用 Lens

↓

生成问题
```

而不是：

```
Lens

↓

直接生成面试
```

同理：

```
Resume Skill

↓

Lens

↓

Evidence

↓

Agent

↓

Resume
```

# 然后我再看 V5，我发现它其实还有几个地方偏了。

---

# 第一个问题：search_evidence 太大了

V5 最大的问题不是工具数量。

而是：

**一个工具承担了五种职责。**

现在：

```
search_evidence()

↓

全文搜索

↓

Requirement Expansion

↓

Decision Trace

↓

Ranking

↓

Relation Query

↓

Pagination
```

已经快变成 SQL 了。

Agent 反而不知道怎么调用。

---

我更喜欢：

```
search()

explore()

trace()

verify()

snapshot()
```

五个动作。

每个只有一种语义。

而不是一个万能 search。

---

# 第二个问题：Requirement Expansion 很奇怪

这是我觉得 V5 最不贴合你需求的一点。

例如：

```
Redis

↓

cache

↓

session

↓

pubsub
```

为什么？

谁规定 Redis 就一定代表 cache？

Redis：

可以：

Rate Limit

Queue

Lock

Bloom Filter

Vector DB

Session

Cache

Pub/Sub

Streams

......

所以：

```
Redis

↓

Cache
```

其实已经开始 Interpretation。

这违反 Constitution。

---

真正应该做的是：

```
Redis

↓

找到所有 Redis Evidence
```

然后 Agent：

> 哦，这里是在做 Session。

或者：

> 哦，这是消息队列。

---

所以：

Requirement Expansion

我建议删。

---

# 第三个问题：Evidence Ranking 我保留一半

V5：

```
Benchmark

Tests

LOC

Complexity

Git

```

其实混在一起了。

这里有两种不同的排序。

第一种：

可信度。

例如：

Benchmark

Test

Documentation

Git

第二种：

重要程度。

例如：

核心模块

入口

高连接度

修改频繁

调用次数

完全不是一个维度。

建议拆开：

```
Credibility Score

Importance Score
```

Agent 自己决定用哪个。

不要合成一个 Score。

---

# 第四个问题：Snapshot 不应该只是压缩

这一点我觉得特别重要。

现在：

```
Snapshot

↓

压缩
```

其实没有帮助 Agent 理解。

真正应该是：

```
Project Snapshot

=

Architecture

+

Entrypoints

+

Core Modules

+

Recent Decisions

+

Evidence Summary

+

Navigation Guide
```

注意最后一项：

> Navigation Guide

例如：

```
Authentication

↓

middleware.ts

↓

jwt.ts

↓

user.ts
```

这比压缩重要太多。

Agent 真正需要的是：

> 我下一步去哪看。

---

# 我甚至建议 Lens 的核心能力改成四种，而不是五个 Tool

不是按照功能分。

而是按照 Agent 的理解流程分。

```
Observe

↓

Explore

↓

Trace

↓

Verify
```

对应：

## Observe

第一次认识项目。

```
snapshot()

architecture()

```

---

## Explore

不断探索。

```
search()

neighbors()

path()

```

---

## Trace

理解为什么。

```
history()

decision()

git()

```

---

## Verify

最后写简历。

```
verify_statement()
```

这样整个系统特别统一。

---

# 最后，我会把整个项目重新定义成一句话

我觉得这是最符合你目标的定位：

> **Project-Lens is a Project Understanding Engine for AI Agents. It reduces the cognitive cost of exploring, understanding, and verifying large codebases. Career-related tasks such as resume generation, JD matching, and interview simulation are implemented as Skills built on top of Project-Lens, not inside it.**

这一定义有几个好处：

* 它把**求职**降级为应用，而不是项目本身。
* 它把**理解项目**提升为核心能力，与你希望“减轻 LLM 理解项目负担、增强 Agent 探索能力”的目标完全一致。
* 它给未来留下了扩展空间：以后可以基于同一个 Lens 做代码审查、Bug 定位、架构分析、知识问答等，而不需要修改底层。

**如果让我参与设计 V6，我不会先讨论目录结构、数据库或实现计划，而是先冻结这份 Constitution。**之后每增加一个 Tool、Schema 或数据库表，都必须回答两个问题：

1. **它是在帮助 Agent 更好地理解项目吗？**
2. **如果删除它，Agent 对项目理解能力会明显下降吗？**

只要有一个答案是否定的，这个设计就应该放到上层 Skill，而不是放进 Project-Lens 本身。这样整个项目会长期保持边界清晰，不会随着功能增加再次演变成一个“大而全”的求职助手。
