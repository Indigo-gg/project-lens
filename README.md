# Project-Lens

**AI Agent 的项目知识层**

> Lens 维护的是"项目知识"，Agent 维护的是"职业知识"。
>
> Lens 回答 **What Exists**（代码里有什么事实），Agent 回答 **What It Means**（这些事实意味着什么）。

**Version 6.0** — 基于宪法重新设计，明确工具职责边界

---

## 项目简介

Project-Lens 是一个将代码库转化为结构化、可查询知识索引的工具——它提取代码中的事实、证据和决策历史，供 AI Agent 理解和推理你的项目。

把它想象成**原始代码与 AI 编程助手之间的"接地层"**。Lens 不用让 Agent 去读成千上万的文件，而是预先将项目处理成一份紧凑的、按证据排序的知识包，Agent 可以直接搜索和验证。

## 为什么需要？

AI Agent 擅长理解意图，但在**项目级上下文**上力不从心：

- 无法一次性在上下文中容纳 10 万行代码
- 不知道哪些函数经过了充分测试，哪些是实验性的
- 无法判断一个关于代码库的声明（比如"我们优化了 Redis 缓存"）是否真的有代码和提交历史支撑

Project-Lens 通过以下方式解决：

1. **提取**代码中的结构化事实（函数、类、模块）
2. **绑定**证据到这些事实（Git 历史、测试、基准测试）
3. **排序**事实的重要性和时效性
4. **导出**紧凑的项目快照，适配 Agent 的上下文窗口
5. **验证**声明是否有真实代码证据支撑

## 六大工具

### 1. observe — 构建索引

扫描项目，建立知识索引：事实（函数、类、模块）、边（关系）和证据（Git 提交、测试、基准测试）。

```bash
lens observe --path ./my-project
```

### 2. explore — 探索项目知识

探索项目知识，提供导航路径。回答"某个功能在项目里怎么走？"

```bash
lens explore "authentication" --category security --limit 10
```

结果按可信度或重要性排序：

```
Credibility = 0.3*基准测试 + 0.3*测试覆盖 + 0.2*文档 + 0.2*Git历史
Importance = 0.4*连接度 + 0.3*修改频率 + 0.3*时效性
```

这意味着 Lens 知道：一个有基准测试、最近有提交、复杂度高的函数，比一个无人问津的工具函数更重要。

### 3. trace — 决策追踪

理解决策历史，追溯变更原因。回答"为什么这样实现？"

```bash
lens trace "Redis" --limit 10
lens trace --fact-id 123
```

### 4. verify — 声明验证

验证一句话是否有真实的代码证据支撑。可以用来检验简历描述、PR 说明或设计文档是否与代码一致。

```
输入:  "优化了 Redis 缓存性能"
输出: { confidence: 0.82, evidence: [...], unsupportedParts: [] }
```

### 5. snapshot — 项目快照

导出整个项目知识包，控制在 50K tokens 以内——足够小，能放进 Agent 的上下文窗口。包含关键事实、近期决策和顶级证据，有意省略完整的提交历史。

```bash
lens snapshot --format compact --max-tokens 50000
```

### 6. render — PDF 渲染

将 JSON 数据通过 Typst 渲染为 PDF。

## 工作原理

### 第一步：提取事实

Lens 使用 [Tree-sitter](https://tree-sitter.github.io/)（通过 WASM，无需原生编译）解析源代码，提取函数、类、接口和模块。每个提取结果成为知识图谱中的一个**事实（Fact）**。

### 第二步：绑定证据

从事实出发，结合 Git 历史和测试文件进行证据增强：

- **Git 提交** → 演进证据（谁改的、多频繁、最近什么时候）
- **测试覆盖** → 质量证据（代码有测试吗？）
- **基准测试** → 性能证据（有性能声明吗？）
- **依赖关系** → 架构证据（连接了什么？）

### 第三步：排序与查询

所有证据通过加权公式评分，存入 SQLite 并启用 FTS5 全文搜索。Agent 查询时拿到的是排序后的结果——而不是原始文件堆砌。

### 第四步：快照与验证

Agent 可以导出紧凑快照获得全局理解，也可以针对具体声明验证代码证据。核心循环：**声明 → 验证 → 信任**。

## 安装

```bash
npm install project-lens
```

## 快速开始

```bash
# 1. 索引你的项目
lens observe --path ./my-project

# 2. 探索项目知识
lens explore "database migration" --limit 5

# 3. 追踪决策历史
lens trace "Redis" --limit 10

# 4. 导出快照供 AI Agent 使用
lens snapshot --format compact

# 5. 启动 MCP 服务器（对接 Claude Code）
lens serve
```

## 与 AI Agent 集成

Project-Lens 通过 [Model Context Protocol](https://modelcontextprotocol.io/) (MCP) 暴露全部 6 个工具。将其接入 Claude Code、Cursor 或任何兼容 MCP 的 Agent：

```json
{
  "mcpServers": {
    "project-lens": {
      "command": "lens",
      "args": ["serve"]
    }
  }
}
```

连接后，Agent 可以搜索项目知识图谱、验证声明是否有代码证据支撑、导出紧凑项目快照——全程无需读取原始文件。

## 设计理念

| 原则 | 含义 |
|:--|:--|
| **Lens 回答 What Exists** | Lens 处理事实：代码结构、提交历史、测试覆盖。它不解释这些事实对求职或业务决策意味着什么。 |
| **Agent 回答 What It Means** | AI Agent 接收 Lens 的事实和证据，在上下文中解读："这个 Redis 优化之所以和 JD 相关，是因为……" |
| **证据优先于观点** | 每个事实都有证据。每个声明都可验证。不猜测，不幻觉。 |
| **紧凑设计** | 快照适配上下文窗口。排序让重要的浮现。更少噪音，更多信号。 |
