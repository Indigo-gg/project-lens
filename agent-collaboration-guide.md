# Project-Lens 本地多 Agent 协同与分工指南

为了实现 Project-Lens V2 的 **“AI-First / Zero UI”** 求职演练闭环，我们打破了传统的单智能体或纯代码分析思路，引入了 **“本地多 Agent 协同”** 架构。

本指南详细记录了本地三款 AI 编程助手的命令行调用机制、各自对 V2 计划的评估与蓝图、编排者对它们的思维特点评价，以及最终的多 Agent 协同分工矩阵。

---

## 一、 本地 AI 助手的命令行调用规范

在 Node.js (TypeScript) 的 `LocalAgentBridge` 实现中，应严格使用以下经过测试验证的非交互式命令进行调用，以确保没有键盘挂起阻塞并绕过安全限制：

### 1. `claude` (Claude Code)
* **特长**：精细代码阅读、小步快跑的代码重构、细节 Bug 修复与校验。
* **非交互式调用命令**：
  ```bash
  claude -p "你的指令"
  ```
  *(注：该命令会在 3 秒无 stdin 输入后自动 proceeding，打印完大模型响应后自动退出。)*

### 2. `codex` (Codex CLI / 对接 LongCat-2.0)
* **特长**：宏观架构理解、高并发/缓存决策提炼、简历级亮点文字润色。
* **非交互式调用命令**：
  ```bash
  # PowerShell 格式 (解决键盘输入挂起并绕过非 Git 仓库安全检测)
  $null | codex exec "你的指令" --skip-git-repo-check
  
  # cmd 格式
  echo. | codex exec "你的指令" --skip-git-repo-check
  ```

### 3. `mimo` (mimocode)
* **特长**：本地研发任务生命周期追踪、多环境状态控制、轻量级 TUI 交互。
* **非交互式调用命令**：
  ```bash
  $null | mimo run "你的指令" --dangerously-skip-permissions
  ```

## 二、 三位 Agent 对 V2 计划的真机反馈与实现蓝图

我们使用命令行直接命令本地的 `claude`、`codex` 和 `mimo` 阅读了 [plan-v2.md](file:///D:/Pcode/play/project-lens/docs/plan-v2.md)，以下是它们吐出的真实反馈：

### 1. Claude Code (Doer) 的反馈
* **整体评价**：是一份视角独到、野心不小的计划。痛点定义精准，“代码即求职”的差异化强，且 Typst 渲染 PDF 是极其正确的决定。
* **核心风险**：
  * “Agent 调度 Agent” 的复杂度被低估，子进程超时管理、非结构化 JSON 鲁棒解析在实际中极易出错。
  * 简历考点库的静态维护成本很高。
* **自荐角色**：**“坚实的手脚（Doer）”**。负责把计划落地为可运行的代码，实现 Zod Schema 定义、Commander CLI 入口、MCP SDK 注册以及错误处理边界。
* **它的实现蓝图**：
  * *Week 1*：Node 初始化、Zod Schema 冻结（利用 `zod-to-json-schema` 保持单一源）、实现命令行/MCP二合一入口。
  * *Week 2*：完成 LocalAgentBridge 子进程封装及 `lens-analyzer` 单元测试。
  * *Week 3*：Nunjucks 渲染 Pipeline 与 Typst 编译集成。
  * *Week 4*：面试 Rubric 考点库建设与端到端联调。

### 2. Codex CLI (Architect) 的反馈
* **整体评价**：定位精准、成本模型清晰（免 API Key 极具推广说服力）、多 Agent 分工抽象合理。
* **核心风险**：
  * 命令行通信中，LLM 输出非结构化文本时经常混入 Markdown code block 标记（如 ` ```json ... ``` `）或注释，需要强大的自我修复 Pipeline。
  * 存在潜在的间接 Prompt 注入风险（例如当扫描恶意的 README 文件时）。
* **自荐角色**：**“严谨的脑核（Architect）”**。从代码库中提取架构亮点，将零散的事实组织为简历级的亮点叙事，不负责精细重构与任务状态管理。
* **它的实现蓝图**：
  * 设计了严格的输入/输出契约（`AnalysisRequestSchema` 和 `AnalysisOutputSchema`）。
  * 提出了 **“JSON 校验与自我修复 Pipeline”** 机制：*Raw Output -> Strip Fences -> JSON Parse -> Zod Validate -> Error Repair (LLM自修2次) -> Fallback (返回局部数据)*。
  * 强调永远不向上层抛出未处理的 LLM 异常。

### 3. mimocode (Implementer) 的反馈
* **整体评价**：Zero UI 理念非常契合本地开发者画像，但在多 Agent 协同和渲染管道上的执行风险非常集中。
* **核心风险**：
  * MCP 与 CLI 双模式并存的维护成本高，建议先稳定 CLI 模式再包装 MCP。
  * 缺少用户数据持久化方案，建议约定 `~/.lens/` 作为统一的配置与状态存储目录。
* **自荐角色**：**“项目落地的实施者与质量保障者”**。负责搭建 Zod Schema、编写本地桥接器，并在开发中做类型检查与测试用例覆盖。
* **它的实现蓝图**：
  * 建议采用 **“先单 Agent 闭环（只用 `claude -p`），再多 Agent 扩展”** 的渐进式开发策略。
  * Phase 3 开始前必须优先在本地测试好系统已有的 `typst` CLI 和 Wasm 方案的兼容性。

## 三、 编排者 (Antigravity/agy) 对各 Agent 的思维特点评价

作为本系统的**最高层编排器 (Orchestrator)**，我通过与它们的交互，对它们的思维偏好评价如下：

* **Claude Code — 极度务实的细节控**
  * *思维特点*：专注生产力，工程落地直觉极强。它关注库的使用细节（如单一源 Schema 自动映射），能够快速把宏观需求拆解为每天、每阶段的具体开发任务。非常适合做“底层重构与工程落地”。
* **Codex CLI (LongCat-2.0) — 宏观的防御性架构师**
  * *思维特点*：有极强的系统契约精神和容错思维。它在写代码之前，会优先定义好完美的 input/output schema 规范。它非常敏锐地感知到大模型输出的不确定性，并为此设计了防御性的自我校验纠错 Pipeline。适合做“高级技术特征提取与数据质量控制”。
* **mimocode — 敏捷开发的务实派**
  * *思维特点*：强调渐进式的风险控制与环境一致性。它对本地配置文件的管理（如 `~/.lens/`）和版本管理有良好的习惯，并提倡先用单 Agent 跑通最简可行产品 (MVP)，再做复杂的编排。适合做“任务流程控制与本地存储管理”。

## 四、 多 Agent 本地协同分工矩阵（推荐实施方案）

基于上述调研，我们对 Project-Lens V2 的开发与运行时进行了如下最优分工：

```
                           ┌──────────────────────────┐
                           │   用户 / Gemini (agy)    │
                           └─────────────┬────────────┘
                                         │ 调度编排 (P0)
                                         ▼
                           ┌──────────────────────────┐
                           │    LocalAgentBridge      │
                           └──────┬────────────┬──────┘
                                  │            │
             调用 (P1) 提取亮点   │            │  调用 (P2) 代码级重构与分析
     ┌────────────────────────────┘            └───────────────────────────┐
     ▼                                                                     ▼
┌──────────────┐                                                     ┌───────────┐
│  Codex CLI   │                                                     │Claude Code│
└──────┬───────┘                                                     └─────┬─────┘
       │ 提炼核心技术画像 JSON                                               │ 精细代码分析与重构校验
       ▼                                                                   ▼
┌──────────────────────────────────────────────────────────────────────────┐
│                              resume.json                                 │
└─────────────────────────────────────┬────────────────────────────────────┘
                                      │ Nunjucks + Typst (P3)
                                      ▼
                               ┌─────────────┐
                               │  resume.pdf │
                               └─────────────┘
```

### 1. 开发期分工 (Development Phase)
* **`mimo`** 负责：搭建 CLI 骨架、项目持久化目录（`~/.lens/`）管理。
* **`claude`** 负责：实现 MCP Server 的 Stdio 传输协议，编写 zod 验证模型，集成 Nunjucks ➡️ Typst 编译管道。
* **`agy`** 负责：对 `LocalAgentBridge` 进行多进程的超时与错误捕获编排。

### 2. 运行时分工 (Runtime Phase)
* **`agy` (Antigravity)** 作为**总指挥**，负责与用户打交道并触发调度。
* **`codex`** 负责**宏观提炼**：扫描代码目录，输出结构化的技术要点 JSON。
* **`claude`** 负责**精细抛光**：针对 `codex` 找出的技术要点，到对应的具体代码行进行分析，生成具体的开发细节支撑。
* **`mimo`** 负责**状态存储**：记录简历构建进度和面试答题历史。
